低优先级设计问题 (保留作为记录)
——————————————————————————————————————————

⏭️  #getPathForSerial 二分搜索在空数组上
  src/registry.js
  
  let high = this.#allocations.length - 1
  
  初始时 allocations 为空，high 为 -1。while (low <= high) 直接退出。
  这是安全的，但不如显式守卫清晰。可优化为：
  
  if (this.#allocations.length === 0) return { expired: false }

⏭️  handleRead 中 params.path 检查可能被误用
  src/handlers.js
  
  if (!params.path) {
    if (serial present) resolve from serial
    else params.path = "." // read directory
  }
  
  如果用户明确传递 path="" 会被解释为无路径。工具声明已正确处理，
  但边界模糊。实际使用中不太可能出现问题。

⏭️  CRLF 检测的 endingOf 函数可简化
  src/text.js
  
  当前实现功能正确，可简化为：
    const m = line?.match(/(\r\n|\r|\n)$/)
    return m ? m[1] : ""

⏭️  toToolResult 的 isDetailed 检查使用松散对象
  index.js
  
  if (typeof result.value === "object" && ...)
  
  此检查是安全的，但可考虑使用符号 key 代替字符串属性以避免冲突。

⏭️  #allocations 数组类型不稳定
  src/registry.js
  
  所有推送操作都使用 { x, count, path } 形状。可提取工厂函数提高
  可维护性，但当前实现已足够清晰。

⏭️  输出始终以换行结束
  src/text.js
  
  为显示目的给每行追加换行符是合理的，即使原始文件最后一行没有换行。
  这是有意的设计选择，不是缺陷。
